# Security & Completeness Audit — User Management Control Panel (Re-audit)

**System under review:** Spring Boot 3.3 / Java 21 control panel that provisions customer subscriptions and issues offline-verifiable Ed25519 JWT licenses (`.lic`) to customer Docker apps via a bundled verifier SDK, plus a React admin UI.
**Scope:** `control-panel-api`, `admin-ui`, `license-verifier`, `license-verifier-spring-boot-starter`, `sample-docker-app`, and supporting infrastructure (docker-compose, migrations, CI, docs).
**Predecessor:** `AUDIT.md` (2026-06-01) — 77 confirmed findings + 93 gaps; verdict ALPHA.

## 0. Remediation Status — 2026-06-13 (REMEDIATED & VERIFIED)

All findings in this report were remediated by a 13-agent parallel fix (package-scoped, disjoint file ownership) and verified by a full `mvn clean verify` on JDK 24: **430 tests pass (0 failures/errors/skipped)** across all four modules (License Verifier SDK 45, Control Panel API 350, Spring Boot starter 30, sample-docker-app 5), up from 260. ~168 source files changed, ~70 test files added/updated, 8 new DB migrations (`18-*.sql`).

- **P0 (2/2 fixed):** MFA-login now requires the signed challenge (email-only fallback removed), is rate-limited/lockout-protected, and records each bad code; MFA-challenge tokens carry `purpose=mfa_challenge` and can no longer be replayed as sessions. Plan "Save permissions" now rejects a null/absent list (400) and binds the UI field, so it can no longer wipe entitlements.
- **P1 (17/17 fixed):** usage + SSO cross-tenant IDORs scoped to the org; `@Version`/guarded-UPDATE/pessimistic-lock close the API-key/license/seat lost-update races; license download is a pure audited read; suspend/cancel cascades to the CRL; KEK rotation covers all encrypted columns + drop-guard; webhook fan-out uses a durable cursor; logout denylists + clears the cookie session; rate-limiter keys on the resolved client IP; Spring Boot bumped to 3.3.13 with a real CVE gate; admin-UI auth/MFA/SSO and the broken management pages work against the post-P0 contracts; offline-verifier starter now has a real test suite.
- **P2/P3:** TOTP replay, Redis revocation wiring, DNS-rebind SSRF re-check, SCIM lifecycle, GDPR audit-PII redaction, API error-mapping (4xx/404/405/415 instead of 500), idempotency replay semantics, actuator/swagger lockdown, forward-headers/HSTS, Redis auth, container hygiene, pagination caps, and the two dead checker beans — all addressed.
- **Deliberately deferred (unchanged):** real Stripe integration, SPA CSRF hardening, multi-region, SOC2 docs. Two additional items deferred per this report's own guidance: the KEK low-entropy KDF (cannot be made backward-compatible without a format change) and the Spring Boot 3.4/3.5 line migration (kept to the 3.3.13 patch to stay green).

The findings below are retained verbatim as the historical audit record; the status above supersedes their "fix" recommendations.

## 1. Executive Summary

This re-audit was run after the P0 and P1/P2 remediation waves recorded in project memory (RBAC privilege-escalation fix, tenant-isolation IDOR family, session/license revocation, MFA/SSO/observability/outbox, licensing-depth/webhooks/GDPR/idempotency, SCIM/billing/key-rotation, and a 260-test green suite). The good news: the *original* P0 breaches largely held. The RBAC self-elevation to `SUPER_ADMIN` is closed, most of the 2026-06-01 cross-tenant IDORs now compose a `TenantAccessChecker` predicate, an Ed25519 CRL revocation channel now exists and is fail-closed once configured, per-request status/authority reload is in place, and the previously-empty test tree now has 260 passing tests (confirmed by a live `mvn clean verify` under JDK 21+).

The bad news: nearly every *new* feature surface added in the depth waves shipped its own alpha-quality defects, and the admin UI is effectively non-functional against its own API. The two most serious confirmed issues are (a) the MFA verification endpoint `/api/v1/auth/mfa/login` is exempt from all rate-limiting and lockout and additionally accepts an email-only fallback with no password check, so an attacker who knows a victim's email can brute-force a 6-digit TOTP into a full session — a complete account-takeover path against MFA-enabled accounts including super-admins; and (b) the admin UI's "Save permissions" button silently wipes every entitlement permission from a plan due to a request field-name mismatch, deterministically corrupting the product's core artifact. Around these sit a fresh cross-tenant IDOR family (usage listing, SSO provider delete/test), a cluster of `@Version`-less lost-update races that can silently un-revoke API keys and licenses or exceed seat caps, broken offline-revocation defaults (revocation is off out of the box and the CRL is rollback-replayable), a webhook fan-out with no durable cursor that silently drops events on any outage longer than 10 minutes, an EOL Spring Boot 3.3.x stack behind a CVE gate that can never fire, and seven separate admin-UI pages that 4xx/500/white-screen because their request/response contracts no longer match the backend after the P0 work. Logout is also a no-op for cookie/SSO sessions, and the Redis-backed session-revocation store is never actually wired in production (a `@ConditionalOnBean` ordering trap), so single-session logout silently runs in-memory.

Overall: the architecture and crypto primitives remain sound, the originally-flagged breaches are remediated, but the breadth of the new feature set has outrun its verification. The control plane is closer to enterprise-credible than at 2026-06-01, yet it is not safe for untrusted-tenant production use until the P0/P1 items below are closed.

### Production-Readiness Verdict: **BETA — not production-ready**

> Improved from **ALPHA**. The 2026-06-01 P0 breaches (RBAC privesc, the original IDOR family, dead revocation, zero tests) are remediated and verified. However the MFA/SSO/billing/webhook/key-rotation depth features and the admin UI carry new critical and high defects — most importantly an MFA brute-force/email-fallback account-takeover path, a deterministic plan-entitlement data-loss bug, a new cross-tenant IDOR pair, concurrency lost-updates that defeat revocation, and a near-totally-broken admin UI. Suitable for a controlled internal/closed beta once P0 is fixed; not for multi-tenant production until P1 is also closed.

## 2. Methodology

Fifteen dimension auditors examined the repository in parallel — RBAC/tenant isolation, authentication/session, licensing core & offline verification, cryptography/key management, input handling/injection, API contract/error handling, data layer/transactions/concurrency, webhooks/billing/SCIM/GDPR, admin-UI security & correctness, test-suite adequacy, operations/config/deployment, architecture/code quality, Redis session-revocation wiring, live dependency/supply-chain scanning, and dynamic verification (a real `mvn clean verify`, JaCoCo harvest, and live error-semantics probing against a fresh Postgres 16 + Redis 7). Every critical/high/medium candidate finding was then handed to an independent adversarial skeptic who attempted to refute it against the source, recording either `confirmed` (skeptic agreed end-to-end), `uncertain` (plausible but unproven), or `unverified` (low severity, not independently re-checked). Severity labels in this report reflect the verifier's calibrated assessment (several candidates were up- or down-graded during verification), not the worst theoretical case. The audit model is **Claude Fable 5**. Auditors read source only except for the dynamic-verification pass; no production systems were touched and no source files were modified.

### P0 — Critical (fix before any deployment, including closed beta)

**P0-1. MFA login endpoint has no rate-limiting/lockout and an email-only fallback → full account takeover** · `control-panel-api/.../auth/AuthController.java:164` (+`common/RateLimitFilter.java:52-65`, `mfa/MfaService.java`)
`mfaLogin()` never calls `loginRateLimiter.isLocked()` at entry (it only records failures, which are never consulted on this path), `RateLimitFilter.shouldNotFilter` protects only `/auth/login` and the two password-reset paths (not `/auth/mfa/login`), and the endpoint is `permitAll`. Worse, when no signed challenge is supplied the subject is resolved by **plain email lookup with no password check** (lines 174-177) and only the 6-digit TOTP is verified — collapsing 2FA to a single factor. With a ~3/1,000,000 per-guess success rate and unlimited unthrottled attempts, an attacker who knows only a victim's email can brute-force a valid code (~333k expected guesses) and `completeSession()` issues a full session JWT/cookie — for any MFA-enabled account including super-admins. *Why it matters:* complete authentication bypass of the strongest control in the system, unauthenticated. *Fix:* enforce `loginRateLimiter.isLocked(email, ip)` at the start of `mfaLogin`; add `/api/v1/auth/mfa/login` to `RateLimitFilter`'s protected set; cap attempts per issued challenge and invalidate the challenge after N failures; **require the signed `mfa_challenge` (proof the password step passed) and remove the email-only fallback.**

**P0-2. Admin "Save permissions" silently wipes all plan entitlement permissions (UI/API field mismatch)** · `admin-ui/src/lib/api.ts:166-168` (+`PlanController.java:95`, `PlanService.replacePermissions`)
The UI posts `{"permissions":[...]}` but the backend DTO is `ReplacePermissionsRequest(permissionCodes)` with no `@NotNull` and Jackson ignores unknown properties, so `permissionCodes` binds to `null`. `replacePermissions` runs `deleteByIdPlanId(planId)` first and then inserts nothing — returning 200 with a "Permissions updated" toast. *Why it matters:* the normal admin flow deterministically deletes every entitlement permission on the plan; licenses issued afterward embed no permissions, corrupting the product's core artifact. Fail-closed security-wise (removes, never grants) but a guaranteed data-loss/correctness defect reachable by a routine click. *Fix:* rename the UI field to `permissionCodes` (or add `@JsonAlias`); reject a null/absent list with 400 instead of treating it as "delete everything"; add a contract test from the UI client shape.

### P1 — High (fix before multi-tenant production)

**P1-1. Cross-tenant IDOR: `GET /subscriptions/{subId}/usage` has no tenant scoping** · `usage/UsageIngestController.java:58`
Gated only by `@PreAuthorize("hasAuthority('subscription.read') or hasAuthority('usage.read')")` with no `TenantAccessChecker` predicate; the service queries purely by subscription id. `usage.read` is org-mintable (`ApiKeyService` creatable-scopes), so any org OWNER/ADMIN can mint a key for org A and read org B's usage events (feature keys, quantities, jti, metadata) and quota limits by passing B's subId. *Fix:* add `and @tenantAccess.canReadSubscription(#subId)` (the checker already exposes `canReadUsageForSubscription(UUID)`), mirroring `LicenseController.list`.

**P1-2. Cross-tenant IDOR: SSO provider delete/test resolve by id only, not scoped to the path org** · `sso/SsoController.java:43-54`
`DELETE /orgs/{orgId}/sso/{id}` and `POST /orgs/{orgId}/sso/{id}/test` only prove the caller is OWNER/ADMIN of the *path* org, then call `service.delete(id)`/`service.test(id)` via `repo.findById(id)` with no `orgId` equality check. An admin of org A can delete or probe another tenant's SSO provider given its UUID. Sibling code (`WebhookController.delete` uses `findByIdAndOrgId`, `ApiKeyService.revoke` checks org equality) does this correctly. *Fix:* scope both operations to `(id, orgId)` via `findByIdAndOrgId`, 404 on mismatch.

**P1-3. MFA login email-fallback bypasses the password factor** · `auth/AuthController.java:170-177`
(Tightly coupled to P0-1; tracked separately because it is the enabler.) Supplying `{email, code}` with no challenge skips the only proof the password step succeeded, reducing 2FA to possession of one TOTP code. *Fix:* require the signed challenge to complete MFA login; reject requests lacking it.

**P1-4. `GET /licenses/{jti}/download` silently mints a brand-new license (new jti, new DB row, new outbox event) with no audit record** · `licenses/LicenseController.java:87`
The signed artifact is served from an in-memory per-instance `LicenseEnvelopeCache`; on any miss (process restart, second instance, expiry eviction) the GET calls `issuer.issue(...)`, minting a new jti and `license_tokens` row and publishing a `LicenseIssued` event. The returned `.lic` jti differs from the URL jti; a GET has unbounded write side-effects; an expired token yields a fresh 1-day license (`max(1, daysRemaining)`); and because audit advice fires only on POST/PUT/PATCH/DELETE, the issuance writes **zero audit rows**. The GET is gated by `canReadLicenseByJti` (any org member or API key), while the explicit POST issue path requires `canManageOrg` and denies API keys — so a read-only principal can mint licenses it is forbidden to issue. *Fix:* persist the issued JWT and make download a pure read returning the stored artifact (404/410 if unavailable); never call `issuer.issue()` from a GET.

**P1-5. Suspend/cancel does not revoke outstanding licenses or reach the CRL; heartbeat ignores subscription status** · `subscriptions/SubscriptionService.java:106`
`suspend()`/`cancel()` only flip status and emit an event that nothing consumes for revocation; `markRevoked()` is invoked only from the manual per-jti endpoint; `ActivationService.heartbeat()` never checks the owning subscription's status. For an offline-license product the CRL is the only enforcement channel, so suspending/cancelling a customer leaves every already-issued license fully valid (and still acquiring seats) until natural expiry. *Fix:* on suspend/cancel, cascade `markRevoked()` to all ACTIVE `license_tokens` of the subscription (or add an outbox consumer) so they land on the CRL; reject heartbeats whose subscription is not ACTIVE.

**P1-6. KEK rotation re-encrypts only signing keys, orphaning MFA/webhook/SSO secrets under the old KEK** · `keys/KeyService.java:178`
One `KeyEncryptor`/KEK protects four secret categories (Ed25519 keys, TOTP secrets, webhook HMAC secrets, OIDC client secrets) but `rotateKek()` walks only `signing_keys`. The versioned envelope embeds the writing KEK id, so once an operator removes the old KEK (the point of rotation), `KeyEncryptor.decrypt` throws for every MFA secret (breaks all MFA logins), webhook secret (every delivery quarantines), and SSO secret — silent, irreversible loss. *Fix:* drive rotation from a registry of every `(table, column)` encrypted by `KeyEncryptor`; add a startup guard refusing to drop a KEK while any blob still references it.

**P1-7. API-key revocation can be silently undone by a concurrent `verify()` full-column update (lost update, no `@Version`)** · `apikeys/ApiKeyService.java:89`
`verify()` runs a write transaction on every API-key request, loading the row and saving `lastUsedAt`; `ApiKey` has no `@Version`/`@DynamicUpdate`, so Hibernate writes a full-column UPDATE that re-persists the stale `revoked_at=NULL`. Under READ COMMITTED, a `revoke()` that commits between a concurrent `verify()`'s read and commit is overwritten — the key is permanently un-revoked. *Fix:* update `last_used_at` via a targeted `UPDATE ... WHERE id=? AND revoked_at IS NULL` (or add `@Version`); make `revoke()` a guarded conditional UPDATE.

**P1-8. License heartbeat can overwrite a concurrent revocation/expiry back to ACTIVE (lost update on `license_tokens`)** · `licenses/ActivationService.java:129`
Same pattern: `heartbeat()` loads the token, checks status, then full-column-saves `lastSeen*`. `LicenseToken` has no `@Version`, so if `markRevoked`/expiry commits between the read and the heartbeat's flush, the heartbeat rewrites `status='ACTIVE', revoked_at=NULL`. The un-revoked jti then drops off the CRL and verifies again offline. Heartbeats are attacker-controlled and unthrottled, so a customer can drive the race. *Fix:* add `@Version` to `LicenseToken` (+migration), or update last-seen via `UPDATE ... WHERE jti=? AND status='ACTIVE'`; make revocation a guarded conditional UPDATE.

**P1-9. Seat-limit enforcement TOCTOU: concurrent first heartbeats from distinct nodes exceed the licensed seat cap** · `licenses/ActivationService.java:88`
For a new node the code counts active seats, compares to the limit, then inserts — with no lock, no serializable isolation, and a unique constraint that only collides on the same `(jti,node_id)`. N concurrent first heartbeats each read `limit-1` and all insert, exceeding the cap by the concurrency level; once over, renewals always succeed so the overage persists. This is the core monetized control. *Fix:* pessimistic-lock the token row before counting, or enforce the cap with a conditional `INSERT ... SELECT WHERE (count) < limit`. (Merges the medium "TOCTOU" and high "race" filings.)

**P1-10. Invoice generation rates ALL lifetime usage with no period scoping and no duplicate-invoice guard** · `billing/BillingService.java:124-177`
`generateDraftInvoice` loads `usageQuotaRepo.findBySubscriptionId` (every period's rows), merges quantities across all of them, and sets the window to global min/max — there is no period parameter and no existing-invoice check, and no unique `(subscription_id, period)` constraint. Each call re-bills lifetime usage; calling generate+issue twice double-bills. Today's `ManualBillingProvider` is a no-op, so no real money moves yet, but the persisted invoices/audit rows are the billing record and the provider is an explicit seam. *Fix:* accept an explicit period and query only matching quota rows; reject/return the existing invoice when one already covers that subscription+period; add the unique constraint.

**P1-11. Webhook fan-out has no durable cursor — events older than the 10-minute lookback are silently, permanently dropped** · `webhooks/WebhookDispatchScheduler.java:138-155` (merges three filings: quality/data/integrations)
`fanOut()` scans `occurred_at >= now() - scanLookback` (default PT10M) every tick with no checkpoint and no per-event "fanned-out" marker; `enqueue()` is the only producer of `webhook_deliveries` rows. Any event that ages out of the window before a successful tick never produces deliveries — no error, no FAILED row, nothing to replay. Reachable via crash-loop/deploy/outage and, worse, self-inflicted scheduler starvation: a single tarpit webhook can stretch fan-out gaps past 10 min and drop events for *all* tenants (the default single scheduler thread is shared and `deliverDueBatch` POSTs up to 100 rows synchronously). *Fix:* track a durable high-water mark (or a `fanned_out_at` column / `FOR UPDATE SKIP LOCKED` claim) and scan from `min(watermark, now-lookback)`; bound the scan with a LIMIT and isolate fan-out on its own scheduler thread. (Reliability defect, not a security vuln, but silent cross-tenant data loss.)

**P1-12. Logout is a no-op for cookie/SSO sessions: jti never denylisted, cookie never cleared, yet 204 returned** · `auth/AuthController.java:229` (merges authn-medium + redis-high filings)
`JwtAuthFilter` accepts the session JWT from the `cp_session` HttpOnly cookie, which is the *only* credential a post-SSO browser receives, but `logout()` reads only the `Authorization: Bearer` header and never emits an expiring `Set-Cookie`. SSO/cookie sessions can therefore never be revoked via logout and the cookie keeps authenticating until JWT expiry (PT30M). *Fix:* resolve the token with the same precedence as `JwtAuthFilter` (Bearer, else `cp_session`), denylist its jti, and always append `Set-Cookie cp_session ... Max-Age=0` on the 204.

**P1-13. License-verifier Spring Boot starter runtime (`CrlRevocationChecker`, `LicenseService`) has ~0% test coverage** · `license-verifier-spring-boot-starter/.../CrlRevocationChecker.java` (merges tests + dynamic-verification; JaCoCo: 0/68 and 1/63 lines)
The fail-closed offline-verification logic every customer app runs — CRL fetch/cache/TTL, stale-beyond-max non-operational, keep-previous-on-refresh-failure, `crl-fail-closed` startup behaviour, and the REVOKED-wins status precedence — is never exercised (the only starter test mocks `LicenseService`; the control-panel IT builds its own inline checker, giving a false sense of coverage). A fail-closed→fail-open regression would ship silently to all consumers. *Fix:* add starter integration tests booting the auto-config against a stub CRL endpoint: never-loaded→deny-all, revoked-jti rejected, stale→non-operational, refresh-failure→previous retained, `crl-fail-closed=true`→startup abort; plus `LicenseService` load/status transitions.

**P1-14. EOL Spring Boot 3.3.4 stack with known-vulnerable managed dependencies, behind a CVE gate that can never fire** · `pom.xml:10` (merges ops + dependency-cve)
The 3.3.x OSS line reached end of support 2025-06-30, so no patches are consumed; the inherited stack is Tomcat 10.1.30, Spring Framework 6.1.13, Spring Security 6.3.3, logback 1.5.8, plus pinned nimbus-jose-jwt 9.40, json-smart 2.5.1, BouncyCastle 1.72 (via OpenSAML 4.3.2, itself EOL), pgjdbc 42.7.4. The OWASP dependency-check job runs `-DfailBuildOnCVSS=11` (impossible; CVSS max is 10) plus `continue-on-error` at job and step level, and there is no Dependabot/Renovate — so vulnerable majors ride along with no signal. *Fix:* bump the parent to 3.3.13 (final 3.3.x; closes the Tomcat/Spring/Security/logback findings) and plan migration to a supported line (3.4/3.5, which also moves SAML off EOL OpenSAML 4); add explicit overrides for json-smart (≥2.5.2), BouncyCastle (≥1.78), pgjdbc (≥42.7.7); set `failBuildOnCVSS` to a real threshold and add an NVD API key + date-stamped cache.

**P1-15. Login/password-reset rate limiter keys on raw `getRemoteAddr()` — collapses to one shared bucket behind the documented proxy** · `common/RateLimitFilter.java:73`
Every other client-IP consumer uses `TrustedProxyResolver`, but `RateLimitFilter` reads the raw peer address and the justifying comment ("nginx sets X-Real-IP") is false — the shipped nginx has no `proxy_pass`. Behind the documented TLS-terminating proxy all clients share the proxy IP, so the single 10-req/min bucket is shared by the entire user base: 10 unauthenticated POSTs/min to `/auth/login` blocks login and both password-reset endpoints for everyone. The bucket map is also an unbounded `ConcurrentHashMap`. *Fix:* key buckets on `TrustedProxyResolver.resolveClientIp()`; bound the map (Caffeine maxSize + expireAfterAccess); fix/delete the nginx comment.

**P1-16 through P1-17 (admin UI functional breaks — security-relevant feature failures).** The admin UI's contracts drifted from the API after the P0 tenant-isolation work; the following pages are confirmed broken end-to-end. They are functional/correctness failures (fail-closed, not exploitable) but render core admin workflows unusable:

- **P1-16. Authentication & SSO are unusable from the only first-party client.** *MFA-enabled admins cannot log in* — the UI has no MFA handling at all, so the `{mfaRequired:true,...}` response is treated as success and bounces back to `/login` silently (`admin-ui/src/lib/auth.tsx:48`). *SSO browser login dead-ends* — after SSO the backend sets only the `cp_session` cookie and redirects, but the SPA short-circuits when localStorage has no token and never attempts a cookie-authenticated `/me` (`auth.tsx:22`). *`/auth/me` and login user-shape mismatch* breaks all client-side permission gating — `hasPermission()` reads `user.permissions/roles` that the login `UserDto` does not contain, so even a super-admin sees only the Dashboard after login (`auth.tsx:66`). *Fix:* add an MFA TOTP step calling `/auth/mfa/login`; bootstrap auth via cookie-authenticated `GET /auth/me` (and on `?sso=success`); unify login and `/me` on one `{user, permissions, orgs}` contract.
- **P1-17. Core management pages 4xx/500/white-screen.** *Global Licenses page + Dashboard stats always fail* — they call `licenses.list()` with no params but the endpoint requires `subscriptionId` (the P0 tenant-leak fix), so the page errors and the dashboard silently reads 0 (`admin-ui/src/pages/LicensesPage.tsx:21`). *API-keys page* never displays the one-time plaintext key (field mismatch `key`/`keyPrefix` vs `plaintext`/`prefix`) and white-screens the SPA when listing keys (`k.scopes.map` on a raw JSON string, no error boundary) (`ApiKeysPage.tsx:202`). *SSO config page* is built against a single flat object while the API is a list of `{type, config}` — every save 400s (`SsoConfigPage.tsx:61`). *Plan management* create/save-features 400 and the edit page + permission picker crash on `Map`/`PagedResponse` shape mismatches (`PlanEditPage.tsx:66`). *Fix:* generate the TS client from the controller DTOs (or share an OpenAPI spec), unwrap `PagedResponse.items`, align field names and `Map` shapes, add a top-level React error boundary, and add an aggregate/scoped endpoint for the global Licenses view.

### P2 — Medium (harden before GA / first enterprise contract)

Grouped by theme; all **confirmed** unless marked.

**Authentication, MFA & session**
- **No TOTP replay protection** (`mfa/MfaService.java:210`): stateless verifier with `allowedTimePeriodDiscrepancy=1` accepts any code for the whole ~90s window; no last-accepted time-step is stored. *Fix:* persist the last accepted step per `UserMfa` and reject `step <= last`.
- **`MfaService` re-implements HS256 against the same session secret — latent MFA bypass** (`mfa/MfaService.java:67`): the challenge is a second Nimbus HS256 token using the same `app.auth.session-secret`. Critically, `SessionTokenService.parse` checks neither issuer nor `purpose`, so a password-only `mfa_challenge` token is structurally a valid session token and can be replayed as a Bearer session for any `tokenVersion==0` user — bypassing TOTP. *Treat at P1 priority alongside P0-1.* *Fix:* one shared `HmacTokenCodec` with explicit purpose/issuer checks on parse; reject non-session tokens in `JwtAuthFilter`.
- **Admin session JWT in `localStorage`** (`admin-ui/src/lib/api.ts:24`): defeats the backend's HttpOnly `cp_session` mitigation; XSS/supply-chain can exfiltrate a super-admin token. *Fix:* rely on the cookie; bootstrap via `/me` with credentials; stop persisting the token in JS.
- **RedisSessionRevocationStore never registered in production** (`auth/RedisSessionRevocationStore.java:27`): `@Component @ConditionalOnBean(RedisConnectionFactory.class)` on a component-scanned bean is evaluated before `RedisAutoConfiguration` registers the factory, so it never matches and the in-memory store is silently used — single-session logout is per-JVM and lost on restart. (Bulk revocation via durable `users.token_version` is unaffected.) *Fix:* wire it like `LoginRateLimiterConfig` using `ObjectProvider<RedisConnectionFactory>`; log which impl is selected; add a Testcontainers-Redis test.
- **Fail-open denylist write** (`auth/RedisSessionRevocationStore.java:55`): a logout during a Redis error is swallowed with a WARN while logout returns 204 — the jti denylist has no durable fallback, so the token stays valid until exp. *Fix:* return 503 (client retries), or degrade to a durable DB token-version bump, or back the denylist with a small DB table.
- **`isJtiDenylisted` fails closed with no Redis command timeout** *[unverified]* (`RedisSessionRevocationStore.java:69`): once the wiring above is fixed, any Redis stall blocks every authenticated request for Lettuce's default 60s. *Fix:* short `spring.data.redis.timeout`, circuit breaker, readiness gating.

**Tenant isolation & authorization governance**
- **Org ADMIN can remove an OWNER** (`orgs/OrgService.java:137`): `removeMember` only blocks the last OWNER, with no actor-vs-target rank check (unlike `addMember`). *Fix:* reject when `rank(target) >= rank(actor)`, super-admin exempt.
- **Four overlapping authorization checker beans, two dead** (`orgs/OrgAccessChecker.java:22`): `@tenantAccess`, `@orgAccess`, dead `@subAccess`, dead `@licenseLookup`, plus hand-rolled checks in `DataPrivacyController`; the `rank()` switch is duplicated and an org-bound API key passes `@tenantAccess.canAccessOrg` but fails `@orgAccess.isMember`. Fail-closed today but drift-prone in security-critical code. *Fix:* delete the two dead beans, fold `OrgAccessChecker` into `TenantAccessChecker`, keep one `rank()`.

**Licensing & offline verification**
- **Revocation is disabled by default in the only shipped consumer and the starter auto-config** (`sample-docker-app/.../application.yml:22`): blank `crl-url` silently wires `RevocationChecker.none()` with no startup warning; a deployed app following the shipped docs performs no revocation checking. *Fix:* default/require a CRL URL when `strict=true`; log a prominent WARN when revocation is off.
- **CRL refresh accepts any validly-signed CRL with no monotonicity check (rollback/replay)** (`license-verifier-spring-boot-starter/.../CrlRevocationChecker.java:104`): `current.set(rl)` unconditionally; a MITM/caching-proxy/mirror can replay an older signed CRL that omits a recently-revoked jti, suppressing revocation for up to ~2h. *Fix:* reject a fetched CRL whose `issuedAt` is older than the cached one.
- **READ_ONLY-on-expiry grace cannot be established at startup** (`license-verifier/.../LicenseVerifier.java:159`): the verifier hard-rejects expired licenses, so a container restart after expiry fails closed (refuse-to-start or NOT_LOADED/deny-all) instead of entering the documented read-only grace. *Fix:* add a verify mode returning an expired-but-otherwise-valid License so the service can enter READ_ONLY at startup.
- **Unauthenticated CRL endpoint re-scans and re-signs on every request** (`licenses/CrlController.java:47`): `permitAll`, loads all revoked tokens, decrypts the signing key, and Ed25519-signs per hit, with no caching and no rate-limit — anonymous DoS amplification that grows with the revoked set. *Fix:* cache the signed CRL JWS keyed by last-revocation timestamp; cache the decrypted active key with invalidation on rotate/compromise.

**Webhooks, billing, SCIM, GDPR**
- **Webhook delivery never re-validates the destination URL through `UrlGuard` (DNS-rebind SSRF)** (`webhooks/WebhookDispatchScheduler.java:301`): validated once at registration, then `URI.create(row.url())` at send time with DNS re-resolved by the JDK client; an org admin can repoint DNS at an internal address. Blind (redirects off, body discarded) and gated to org admins, but reaches internal HTTPS services. *Fix:* re-run `UrlGuard.resolveAndCheck` before each delivery and pin the connection to the vetted IP (mirror `UrlGuard.fetchPinned`).
- **Subscription state machine unguarded: CANCELLED is not terminal** (`subscriptions/SubscriptionService.java:106`): `cancel → suspend → reactivate` resurrects a cancelled subscription (only `reactivate` checks for CANCELLED), which then mints licenses again. *Fix:* enforce an explicit allowed-transition matrix; reject illegal transitions with 409.
- **SCIM soft-delete breaks the deprovision/re-provision lifecycle and DELETE conformance** (`scim/ScimService.java:269-280`): DELETE keeps the mapping, so GET still returns the "deleted" resource (200, not 404) and a re-create POST hits a permanent 409. *Fix:* treat a deleted mapping as absent in get/list/provision (allow re-link), or truly delete.
- **GDPR erasure leaves raw IP and raw email in `audit_log`** (`compliance/ErasureService.java:94-98`): the "audit writer already masks emails" justification is false — `AuditWriter` serializes payloads verbatim and always stores the client IP, and `ScimService` writes the raw email into the provisioned-event payload. *Fix:* redact PII-bearing audit fields for the erased subject within the erasure transaction, or guarantee no raw PII is written at audit time.
- **Webhook config knobs `app.webhooks.max-attempts` / `timeout` are dead** (`webhooks/WebhookDispatchScheduler.java:69`): hardcoded `MAX_ATTEMPTS=8` and 3s/5s timeouts; the documented PT10S timeout is wrong and tuning the properties is a silent no-op (and the public OpenAPI repeats the fiction). *Fix:* bind the properties (or delete the dead YAML + correct the docs).
- **UI never uses the Idempotency-Key support; create-and-issue allows duplicate subscriptions** (`admin-ui/src/pages/SubscriptionCreateWizard.tsx:89`): no UI mutation sends the header, CORS `allowedHeaders` omits `Idempotency-Key`, and the wizard's buttons re-enable mid-flight so a double-click creates a duplicate subscription + license (no DB uniqueness guard). *Fix:* generate a per-submission key, add the header to CORS, keep buttons disabled until the full sequence settles.

**Data layer, transactions & concurrency**
- **Usage quota limit enforcement race** (`usage/UsageIngestService.java:112`): plain SELECT check then atomic upsert; concurrent batches both pass and overrun the limit. The provided `findForUpdate` (PESSIMISTIC_WRITE) is never called. *Fix:* lock the quota row or enforce the limit inside the upsert (`DO UPDATE ... WHERE consumed+excluded <= limit`); catch the dedup constraint to return idempotently.
- **`getQuotaStatus` single-result query against a key that accumulates one row per month** (`usage/UsageIngestService.java:185`): `findBySubscriptionIdAndFeatureKey` returns `Optional`, so after the second usage period it throws `IncorrectResultSizeDataAccessException` → 500. *Fix:* return a list (or query the current period).
- **Invoice issue/void race, no `@Version`** (`billing/BillingService.java:184`): unlocked status check then provider call then save; concurrent issue double-charges (latent until a real provider), and a concurrent VOID is overwritten back to ISSUED. *Fix:* add `@Version`/row lock; flip status before the provider call (or via outbox); make provider calls idempotent.
- **`@Version` conflicts surface as raw 500s; every login writes the versioned users row** (`common/GlobalExceptionHandler.java:103`): no handler for `ObjectOptimisticLockingFailureException`, and `completeSession` stamps `lastLoginAt` on the versioned `User` every login. *Fix:* map optimistic-lock failures to 409 with retry hint; stamp last-login via a non-versioned targeted UPDATE.
- **Check-then-insert races on unique constraints surface as 500s** (`billing/BillingService.java:71` and `OrgService.addMember`, `ScimService.provision`, `UserService.createUser`, `PlanService.createPlan`): no `DataIntegrityViolationException` handling; SCIM additionally lacks a `UNIQUE(org_id, user_id)` and silently creates duplicate mappings. *Fix:* catch the violation → 409/idempotent; add the SCIM unique constraint.
- **`outbox_events` grows unbounded with no purge; webhook fan-out re-scans it every 5s with no usable index** (`webhooks/WebhookDispatchScheduler.java:139`): `findSince` over all statuses uses neither partial index → sequential scan ~17k×/day; `webhook_deliveries` and `idempotency_keys` similarly never purged (the TTL-cleanup helper has no caller). *Fix:* add a b-tree index on `occurred_at` (or a durable cursor), add retention jobs, replace the `license.expiring` count-dedup with a column.
- **WebhookDispatchScheduler `@Transactional` self-invocation** (`webhooks/WebhookDispatchScheduler.java:120`): `@Scheduled dispatch()` self-invokes package-private `@Transactional` methods, so the `FOR UPDATE SKIP LOCKED` claim runs autocommit and locks release immediately → duplicate deliveries across instances. *Fix:* route through a self-injected proxy (as `AuditWriter` does) or a separate transactional bean. (Reliability; at-least-once with `X-CP-Delivery` idempotency key partly mitigates.)
- **LicenseLifecycleScheduler self-invocation** (`licenses/LicenseLifecycleScheduler.java:62`): same proxy-bypass; expiry transition and outbox event are not atomic, and concurrent sweeps emit duplicate `license.expired/expiring` events. *Fix:* route through the proxy; do the expiry as a single guarded set-based UPDATE with outbox insert in the same tx; dedupe warnings with a unique marker.
- **All assigned-UUID entities without `@Version` trigger `merge()`+SELECT on every save** (`usage/UsageIngestService.java:128`): a 100-event ingest batch issues 200 statements (no batching configured). *Fix:* implement `Persistable<UUID>` on a mapped superclass (or add `@Version`); set `spring.jpa.open-in-view=false`.

**API contract & error handling**
- **Catch-all `@ExceptionHandler(Exception.class)` maps standard client errors to 500** (`common/GlobalExceptionHandler.java:103`, confirmed live): malformed JSON, wrong method/content-type, type-mismatched UUID path vars, missing required params, unknown subpaths all return 500 instead of 400/404/405/406/415 — contradicting the OpenAPI contract and polluting 5xx SLOs/alerts. *Fix:* extend `ResponseEntityExceptionHandler` (or add explicit handlers) and keep `Exception.class` as last-resort 500.
- **Unauthenticated requests to protected endpoints return 302 to the OIDC authorization endpoint instead of 401** (`auth/SecurityConfig.java:155`, confirmed live): SSO-enabled-by-default installs the OAuth2 login entry point on the stateless API chain, so credential-less API/CLI/metrics calls get a browser redirect (a garbage Bearer correctly gets 401 JSON — internally inconsistent). *Fix:* set an explicit 401 `authenticationEntryPoint` for `/api/**` and `/actuator/**`; scope `oauth2Login`/`saml2Login` to the SSO paths in a separate filter chain.
- **Idempotency layer caches 4xx client-error responses** (`common/IdempotencyInterceptor.java:140`): 400/404/409/422 are persisted for the TTL (default P1D) and replayed, pinning transient errors and blocking corrected retries under a stable key. *Fix:* only cache 2xx (and arguably 3xx); release the claim for 4xx so a corrected/later retry can re-execute.
- **List endpoints advertised as paginated have no page/size and no upper bound; the outbox feed is fully unbounded** (`events/EventStreamController.java:26` and Org/RBAC/License/Billing/Subscription/Sso/ApiKey lists): `findSince` has no LIMIT, so `event.read` can pull the entire `outbox_events` table; several endpoints fabricate `PageMeta` over full `findAll()`. *Fix:* enforce server-side page/size (reuse `PageRequestParams.MAX_SIZE`), cap `findSince` with a LIMIT, and either implement real pagination on the org/RBAC endpoints or stop returning a paged envelope. (Merges the data and quality "fake pagination" filings.)
- **No request-body size limit on `application/json` endpoints** *[unverified]* (`application.yml:24`, observed live: a 3 MB body to `/auth/login` was fully parsed): multipart/form limits don't cover JSON, so unauthenticated large-JSON POSTs to public auth endpoints force full buffering + Jackson parsing. *Fix:* a small `OncePerRequestFilter` rejecting oversized Content-Length with 413, or a reverse-proxy body limit.

**Input handling**
- **Leftmost `X-Forwarded-For` token trusted verbatim** (`common/TrustedProxyResolver.java:70`): when the peer is a trusted proxy the leftmost (client-controlled, with the standard append config) token is returned with no IP-syntax validation — enabling audit/activation source-IP spoofing and, for a non-IP value, `?::inet` cast failures that silently drop fail-open audit rows or roll back fail-closed mutations. *Fix:* parse XFF right-to-left skipping trusted hops, validate the chosen token is a well-formed IP, fall back to the socket peer.

**Operations & deployment**
- **Observability config is dead** (`application.yml:170`): `app.observability.access-log.enabled` and `tracing.enabled` are read by no code — there is no HTTP access log and no cross-hop trace propagation, yet the YAML claims both "enabled by default in prod." *Fix:* implement the access-log filter and propagate `requestId` into the outbox/scheduler MDC, or delete the dead properties.
- **No metrics on any critical flow** (`common/ObservabilityConfig.java:56`): `TimedAspect` is wired but `@Timed`/Micrometer counters are used nowhere; outbox-FAILED, webhook-FAILED, Redis fallbacks, and login lockouts are log-only, so no alert can be built on them. *Fix:* add counters/gauges for outbox depth/FAILED, webhook attempts/failures, licenses issued/revoked, login failures/lockouts, Redis fallbacks; `@Timed` on issuance and dispatch.
- **`/actuator/metrics` and `/prometheus` readable by any authenticated principal** (`auth/SecurityConfig.java:130`): only `.authenticated()`, and `ApiKeyAuthFilter` authenticates on every path — any tenant user or customer API key can pull the full scrape (URI inventory, latencies, error rates, JVM internals). *Fix:* require a dedicated authority/super-admin for `/actuator/**` beyond health/info; exclude `/actuator` from the API-key filter; or a separate management port.
- **Swagger UI and full OpenAPI spec exposed unauthenticated with no prod kill-switch** (`auth/SecurityConfig.java:117`): springdoc is permitAll and never disabled in any profile, handing anonymous clients the complete enumerated API surface. *Fix:* disable springdoc outside dev, or require auth for `/swagger*` and `/v3/api-docs*`.
- **No forward-headers handling: the configured HSTS header is never emitted behind the documented TLS-terminating proxy** (`application.yml:29`): no `server.forward-headers-strategy`/`ForwardedHeaderFilter`, so `request.isSecure()` is false and Spring Security never writes HSTS (and scheme-derived URLs resolve as http). *Fix:* set `server.forward-headers-strategy=framework` and require the trusted proxy to set `X-Forwarded-Proto/For`.
- **Redis runs unauthenticated on the compose network while backing session revocation and lockout counters** (`docker-compose.yml:19`): no `requirepass`/ACL; any container (incl. the admin-ui dev container running `npm install`) can `FLUSHALL`, resetting brute-force counters and reviving logged-out jti denylist entries (bulk revocation via DB token-version is unaffected). *Fix:* set a Redis password from the same required-env pattern as `POSTGRES_PASSWORD`; isolate the dev UI container on a separate network.

**Dependency CVEs (reachability uncertain unless noted)**
- **Tomcat 10.1.30 multiple Important CVEs incl. reachable multipart DoS** (`control-panel-api/pom.xml:20`, confirmed): the reachable vector is the multipart-flood DoS (mitigated by the 1 MB request cap + auth-endpoint rate limit); RCE/HTTP-2 CVEs are not reachable in this config. *Fix:* Tomcat ≥10.1.42 (via Boot 3.3.13). 
- **spring-security 6.3.3 CVE-2025-22228 (BCrypt >72-char bypass)** *[unverified]* — mitigated by the app's 72-char password policy cap; bump to ≥6.3.8 anyway.
- **spring-framework 6.1.13 (RFD CVE-2025-41234, path-traversal CVE-2024-38819)** *[unverified]* — bump to ≥6.1.21.
- **BouncyCastle 1.72 via OpenSAML (DoS/timing/LDAP CVEs)** *[unverified]* and **OpenSAML 4.3.2 EOL** *[unverified]* — pin BC ≥1.78; plan migration to OpenSAML 5 (Boot 3.4+).

**Test coverage gaps (regression risk, not live vulns)**
- **API-key authentication, verification, and revocation are entirely untested** (`apikeys/ApiKeyAuthFilter.java:36`): the real filter is never driven end-to-end (tests inject a synthetic principal), so revoked/expired/malformed/cross-org branches are unverified (JaCoCo: filter 33%, `ApiKeyAuthentication` 0%). *Fix:* an `ApiKeyServiceTest` + an IT driving real `Authorization: ApiKey` headers.
- **Login rate limiter / lockout is completely untested** (`auth/AuthController.java:117`) and the production **`RedisLoginRateLimiter` is 0% covered** *[unverified]* (test profile excludes Redis). *Fix:* unit test the threshold/decay; Testcontainers-Redis IT for lockout + reset.
- **Webhook fan-out / `resolveOrg` (the cross-tenant routing step) untested** (`webhooks/WebhookDispatchScheduler.java:139`): a routing bug could deliver one tenant's events to another's endpoint undetected. *Fix:* tests seeding subscription/license_token events asserting deliveries go only to the correct org.
- **Multiple mutating/admin endpoints have no authz/tenant-isolation tests** (`subscriptions/SubscriptionController.java:71`): subscription overrides, transfer-owner, plan mutations, SSO create/delete/test, audit reads, events — guarded only by `@PreAuthorize` annotations with no negative tests. *Fix:* focused denied-actor/wrong-org tests for the highest-risk endpoints.

**Architecture/quality (no runtime/security impact, maintainability risk)**
- **Audit trail is an implicit four-path ThreadLocal protocol** (`common/AuditContext.java:51`): correctness relies on conventions in comments; side-effecting GETs (license download re-issue) audit nothing. *Fix:* a single `@Audited` annotation/decorator owning set/record/markRecorded/clear across all verbs.
- **`SsoSuccessHandler` is a 320-line transaction-script with no transaction boundary** (`sso/SsoSuccessHandler.java:78`): JIT user/identity/membership writes each commit independently; a failure mid-sequence leaves partial provisioning (self-heals next login). *Fix:* extract an `@Transactional SsoProvisioningService` + a shared `SessionCookieIssuer`.
- **Transaction-script controllers bypass the service layer (Auth, Webhook, RBAC)** (`auth/AuthController.java:111`): inconsistent with the rest of the codebase and already caused duplicated provisioning logic. *Fix:* introduce `AuthService`/`WebhookService`; move RBAC persistence into the service.

## 4. Known Deliberate Deferrals (tracked, not counted as findings)

Per project memory, the following were intentionally left out of the implementation waves and are **not** counted as findings here. They remain open and should be tracked:

1. **Real payment-provider integration (Stripe).** `ManualBillingProvider` is a log-only no-op; `recordCharge`/`finalizeInvoice` do not move money or call an external API. (Note: P1-10 and the invoice issue/void race become higher-impact the moment a real provider is wired at this seam.)
2. **SPA CSRF hardening.** CSRF is disabled server-side and the SPA carries no CSRF token; the cookie is `SameSite=Lax`. Deferred deliberately and not reported as a finding.
3. **Multi-region / data-residency.** No multi-region topology, residency controls, or cross-region replication.
4. **SOC2 / ISO control-mapping docs.** Compliance control mapping, SIEM export, and formal audit-retention policy are not implemented.

## 5. Recommended Remediation Order

**Stage 0 — Block account takeover and data loss (P0, do first; small, targeted changes):**
1. **P0-1 / P1-3** — Throttle `/auth/mfa/login` (lockout + RateLimitFilter), cap attempts per challenge, and remove the email-only fallback (require the signed challenge). Also fix the related `MfaService`/`SessionTokenService` purpose-claim gap (P2) so an `mfa_challenge` token cannot be replayed as a session — these three together restore MFA integrity.
2. **P0-2** — Fix the plan-permissions field mismatch and reject null permission lists; add a contract test.

**Stage 1 — Close the new security boundary regressions (high-value, mostly small):**
3. **P1-1 / P1-2** — Add the tenant predicate to the usage endpoint; scope SSO delete/test to `(id, orgId)`.
4. **P1-12** — Make logout work for cookie/SSO sessions (denylist the cookie's jti + clear the cookie); **P2 wiring** — fix `RedisSessionRevocationStore` registration so single-session logout is actually durable.
5. **P1-7 / P1-8 / P1-9** — Eliminate the lost-update races: targeted/guarded UPDATEs and `@Version` on `ApiKey`, `LicenseToken`, and a lock/conditional-insert for seat enforcement. These directly defeat revocation and the monetized seat cap.
6. **P1-4 / P1-5** — Make license download a pure audited read; cascade suspend/cancel to CRL revocation and reject heartbeats for non-ACTIVE subscriptions.
7. **P1-6** — Make KEK rotation cover all encrypted columns and add the drop-guard.
8. **P1-15** — Key the rate limiter on the resolved client IP and bound the bucket map.

**Stage 2 — Restore the product's enforcement & reliability guarantees:**
9. **Offline revocation defaults & integrity** — enable/require CRL by default (P2), add CRL monotonicity (P2), cache the CRL endpoint (P2), and fix READ_ONLY-on-expiry startup (P2).
10. **P1-11 / P2 outbox** — durable webhook fan-out cursor + retention/index on `outbox_events`; fix the scheduler self-invocation proxy bugs.
11. **P1-13** — add the starter offline-verification test suite (highest-leverage tests in the repo).

**Stage 3 — Platform credibility (GA prerequisites):**
12. **P1-14** — bump to Spring Boot 3.3.13 (+ explicit json-smart/BouncyCastle/pgjdbc overrides), make the CVE gate real, add Dependabot; plan the move to a supported Boot line.
13. **P1-16 / P1-17** — make the admin UI functional: shared generated client/contract, MFA + cookie-SSO login, error boundary, and the broken pages.
14. **API contract & ops** — fix the 500-shadowing exception handler and the 302-instead-of-401 entry point; lock down actuator metrics/swagger; add forward-headers/HSTS; authenticate Redis; wire real metrics and an access log.
15. **P2/P3 data-integrity & validation** — quota/invoice races, state-machine guards, SCIM lifecycle, GDPR audit-PII redaction, check-then-insert 409 mapping, and the remaining validation/pagination gaps.
16. **P3 cleanup** — dead code, helper consolidation, DB invariants, image/build hygiene, and the tracked deferrals when their dependencies (real billing provider, etc.) come online.

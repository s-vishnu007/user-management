## [SEC-001] Privilege escalation: any holder of user.write can assign the SUPER_ADMIN role (including to self, globally)

- **Severity:** CRITICAL
- **Dimension:** authz-rbac
- **File(s)/location:** `control-panel-api/src/main/java/com/example/cp/rbac/RbacController.java` â€” `assignRole()`, `POST /api/v1/rbac/users/{userId}/roles`, lines 50-70 (`@PreAuthorize("hasAuthority('user.write')")`). Seed data: `02-rbac.sql` lines 72, 74-78, 80-84; `12-additional-permissions.sql` lines 21-25. Authority resolution: `PermissionService.permissionsFor()` lines 28-50. Login/resolution: `AuthController.login` line 105, `JwtAuthFilter` lines 60-68.

- **Description:** The role-assignment endpoint is gated only by `hasAuthority('user.write')`. It accepts an arbitrary `roleCode` from the request body (`AssignRoleRequest.roleCode`) and an attacker-controlled `orgId`, resolves the role by code (`roleRepository.findByCode(body.roleCode())`) and inserts a `user_roles` row with no restriction on WHICH roles may be granted and no check that the actor outranks the granted role. Per seed data, `SUPER_ADMIN` is cross-joined to EVERY permission (`02-rbac.sql` line 72: `SELECT r.id, p.id FROM roles r CROSS JOIN permissions p WHERE r.code = 'SUPER_ADMIN'`, reinforced by `12-additional-permissions.sql` lines 21-25), and `ORG_ADMIN`/`ORG_OWNER` are seeded with `user.write` as GLOBAL (org_id NULL) permissions. `PermissionService.permissionsFor()` walks `user_roles -> role_permissions -> permissions` with no filter on role code (`rp.roleId = ur.roleId and ur.userId = :userId and ur.orgId is null`), so a user assigned the SUPER_ADMIN role (even a global orgId=null assignment, which is fully attacker-controllable) receives ALL fine-grained permissions on next login. There is no self-assignment guard: the `userId` is a caller-chosen path variable (only constraint is the `countAssignment` dedup at line 57). Net effect: any org admin/owner (or anyone holding `user.write`) can grant themselves or an accomplice SUPER_ADMIN and obtain platform-wide authority. Note: assigning the SUPER_ADMIN role row does NOT flip the `User.super_admin` boolean column (`User.java` line 47), so the boolean-gated bypasses (`OrgAccessChecker` lines 25/32, the `AuthoritiesLoader.superAdmin` shortcut) are not directly triggered â€” but this does not mitigate the finding, since the attacker still obtains every fine-grained permission code via `permissionsFor`, and those codes are exactly what gate the sensitive endpoints.

- **Attack scenario:** An org administrator (granted ORG_ADMIN, which seeds global `user.write`) calls `POST /api/v1/rbac/users/{theirOwnUserId}/roles` with body `{"roleCode":"SUPER_ADMIN"}` and no `orgId` (global). The insert succeeds because only `user.write` is required. They log out and back in; authorities are resolved through the SUPER_ADMIN role (cross-joined to all permissions), granting them `key.rotate`, `license.issue`/`revoke`, `plan.write`, `audit.read` and every other capability across all organizations.

- **Recommended fix:** Gate this endpoint with a dedicated, higher privilege than `user.write` (e.g. a new `role.assign` / `rbac.admin` permission held only by platform admins), and explicitly forbid assigning the SUPER_ADMIN (and other roles where `role.is_system` is true) via this API. Add an authorization rule that the actor cannot assign a role conferring permissions the actor does not already hold (no privilege amplification), reject self-assignment of elevated roles, and validate/authorize the `orgId` scope (a global orgId=null assignment must require platform-level authority; an org-scoped assignment must require admin rights on that specific org). Audit every assignment with actor and granted role.

## [SEC-002] Revocation is never enforced â€” offline verifier/starter contains zero revocation logic, so a revoked license keeps working until natural expiry (up to 365 days)

- **Severity:** CRITICAL
- **Dimension:** license-revocation
- **File(s)/location:** `license-verifier/src/main/java/com/example/licenseverifier/LicenseVerifier.java` â€” `verifyJwt()` lines 103-149 (validates sig/exp/nbf/aud/iss/kid only); `license-verifier-spring-boot-starter/src/main/java/com/example/licenseverifier/spring/LicenseService.java` â€” `status()`/`readAndVerify()` lines 89-125; `license-verifier-spring-boot-starter/src/main/java/com/example/licenseverifier/spring/RequiresPermissionAspect.java` â€” `check()` lines 21-45. Server side: `LicenseRevocationService.markRevoked()`, `CrlController` `GET /api/v1/licenses/revoked` (permitAll). TTL: control-panel `application.yml` `app.licenses.default-ttl-days: 365`; `LicenseClaimsBuilder.build()` lines 65-68. Integration guide: `integration-guide.md` section 3.2 steps 1-10.

- **Description:** The control panel can mark a license REVOKED (`LicenseRevocationService.markRevoked`) and exposes a CRL endpoint (`CrlController GET /api/v1/licenses/revoked`), but NOTHING consumes it. Grepping the entire verifier lib, the Spring Boot starter, and the sample-docker-app for `revok|crl|revoc` returns zero matches. `LicenseVerifier.verifyJwt()` only checks alg==EdDSA, typ, kid presence, signature, temporal claims (exp/nbf), audience, and issuer â€” no jti blocklist, no CRL lookup. `toLicense()` parses jti but never uses it for revocation. `LicenseService.status()` derives status purely from file presence (NOT_LOADED) and expiry (EXPIRED/READ_ONLY/ACTIVE); `readAndVerify()` only reads the file and calls `verifier.verify()`; `reload()` re-reads on a 24h schedule. `RequiresPermissionAspect.check()` â€” the gate for every `@RequiresPermission` operation â€” branches only on `status()` and `hasAccess()` (permission membership). `LicenseVerifierAutoConfiguration` wires no CRL client; `LicenseProperties` exposes no CRL/revocation-URL field. The default TTL is 365 days, so revocation is effectively a no-op for up to a year. This is systemic, not just JVM: the integration-guide verification recipe and the Node/Python/Go samples end at "Cache claims in memory. Gate endpoints on claims.permissions[]" with no CRL step, so every documented integration path ignores revocation. The CRL endpoint is referenced only as a doc link, never wired into verification.

- **Attack scenario:** A customer's subscription is terminated for non-payment, or a license is leaked/abused. An operator calls `POST /api/v1/licenses/{jti}/revoke`; the DB row flips to REVOKED and a `LicenseRevoked` outbox event is written. The customer's container keeps the same `/etc/app/license.lic` file. On the next 24h reload, `LicenseService` re-reads that file, the JWT still verifies (valid signature, not expired), and every `@RequiresPermission` check continues to pass. The revoked customer retains full paid functionality for the remaining license lifetime (up to ~365 days) with no way for the vendor to cut them off short of rotating the signing key (which breaks ALL customers).

- **Recommended fix:** Make the verifier/starter actually consume revocation. Add an `app.license.crl-url` (or reuse the refresh-from-url base) property and a scheduled component in the starter that fetches the signed CRL, caches the revoked jti set, and have `RequiresPermissionAspect`/`LicenseService` reject the current license when its jti is in that set (fail-closed when the CRL cannot be refreshed and is stale). For high-value entitlements, prefer short-lived licenses (hours/days) with mandatory online renewal so revocation takes effect within one renewal window; 365-day offline tokens with no revocation channel are not safely revocable.

## [SEC-003] Broken org isolation (IDOR): any org admin can revoke any other org's API key

- **Severity:** HIGH
- **Dimension:** apikey-security
- **File(s)/location:** `control-panel-api/src/main/java/com/example/cp/apikeys/ApiKeyController.java` â€” `revoke(orgId, id)` lines 46-51 (`@DeleteMapping("/{id}")`, `@PreAuthorize("hasAuthority('apikey.write') or @orgAccess.isOwnerOrAdmin(#orgId)")`); `ApiKeyService.java` â€” `revoke(UUID id)` lines 84-93. Related: `OrgAccessChecker.java` lines 29-45; `ApiKey.java` lines 31-32 (orgId column); `ApiKeyRepository.java` line 13 (`findByOrgId` exists but unused in revoke).

- **Description:** The DELETE endpoint `/api/v1/orgs/{orgId}/api-keys/{id}` is guarded by `@PreAuthorize("hasAuthority('apikey.write') or @orgAccess.isOwnerOrAdmin(#orgId)")`, which only authorizes the caller as an admin of the `{orgId}` in the PATH. The actual revocation, `service.revoke(id)`, looks the key up by primary key alone (`repo.findById(id)`) and never checks that the resolved `ApiKey.orgId` equals the path `orgId`. The `orgId` path variable is used ONLY for the authorization check and is never passed to the service. An admin of org A can therefore call `DELETE /api/v1/orgs/{A}/api-keys/{some-key-id-belonging-to-org-B}` and successfully revoke org B's key â€” the authorization check passes (caller is admin of A) but the operation mutates a row belonging to B. Sibling operations are correctly scoped (`create` writes under the path orgId; `listForOrg` filters by orgId), so only `revoke` breaks tenant isolation. The global `apikey.write` authority is cross-org by design; the unintended escalation is specifically the per-org admin branch combined with the unscoped `service.revoke(id)` lookup.

- **Attack scenario:** A customer who is OWNER/ADMIN of their own org enumerates or guesses an API key UUID belonging to a competitor's org (UUIDs may leak via the list DTO, support tickets, logs, or be brute-forced over time). They issue `DELETE /api/v1/orgs/{their-own-org}/api-keys/{victim-key-id}`. The PreAuthorize passes because they are admin of their own org, and the victim's production API key is revoked, causing an outage for the competitor's licensed Docker apps that authenticate with that key. The need to learn/guess the victim key UUID is the only practical limiter (consistent with high, not critical).

- **Recommended fix:** Scope the lookup/mutation to the org: change `ApiKeyService.revoke` to accept `orgId` and load with a repository method like `findByIdAndOrgId(id, orgId)` (or verify `k.getOrgId().equals(orgId)` before mutating, throwing notFound otherwise). Apply the same orgId filter to any future get-by-id. Return 404 (not 403) when the key is not in the caller's org to avoid an existence oracle.

## [SEC-004] Failed and denied security-relevant actions are never audited (login failures, lockouts, authorization denials)

- **Severity:** HIGH
- **Dimension:** audit-integrity
- **File(s)/location:** `control-panel-api/src/main/java/com/example/cp/audit/AuditInterceptor.java` â€” `@AfterReturning` advice `afterMutating` (lines 50, 73; sole call site of `AuditWriter.record`). Also: `SecurityConfig` `@EnableMethodSecurity`/`@PreAuthorize` on privileged endpoints (e.g. `KeyController.java:23`, `LicenseController.java:98`, `RbacController.java:51,73`) with no accessDeniedHandler/authenticationEntryPoint; `JwtAuthFilter` lines 55-58; `LoginAttempt` in-memory `ConcurrentHashMap` line 19; `08-audit.sql` lines 27-33 (immutable triggers).

- **Description:** Audit rows are written only by the AOP advice annotated `@AfterReturning`, which by definition fires solely when the controller method returns normally. Every security-relevant failure throws instead of returning: `AuthController.login` throws `ApiException.unauthorized` on lockout (line 86), unknown email (91), inactive account (96), and bad password (98-101) BEFORE `AuditContext.set("auth.login")` (line 112) is ever reached; `@PreAuthorize` denials throw `AccessDeniedException`; failed key rotation / license issuance throw `RuntimeException`/`ApiException`. `GlobalExceptionHandler` catches `AccessDeniedException`, `AuthenticationException` and `ApiException` and returns a `ProblemDetail` without writing any audit entry (it has no `AuditWriter` dependency). There is no `@AfterThrowing`/`@Around` advice anywhere in the audit package. `SecurityConfig` configures no accessDeniedHandler/authenticationEntryPoint, so `AccessDeniedException` denials are silently unaudited; `JwtAuthFilter` writes a `ProblemDetail` on bad/expired tokens without auditing. `LoginAttempt` stores failure/lockout counters in an in-memory `ConcurrentHashMap` (line 19) with no persistence â€” lost on restart, never written to `audit_log`. The `immutable append-only` premise is real (`08-audit.sql` triggers block UPDATE/DELETE). Net result: no record exists anywhere for failed logins, account lockouts, or authorization denials on privileged endpoints (`key.rotate`, `license.revoke`, `rbac.role.assigned`, etc.). High (not critical) because it is a detection/forensics/compliance gap rather than a direct breach of confidentiality or integrity.

- **Attack scenario:** An attacker performs credential stuffing / password spraying against `/api/v1/auth/login` and probes privileged endpoints (`POST /api/v1/admin/keys/rotate`, `POST /licenses/{jti}/revoke`) with a low-privileged token. Every failed attempt and every authorization denial returns a clean 401/403 and leaves zero rows in `audit_log`. Incident responders reviewing the "immutable, append-only" audit log see no evidence of the attack, and the in-memory lockout counter is wiped on the next deploy/restart.

- **Recommended fix:** Audit failures explicitly. Add `@AfterThrowing` advice to `AuditInterceptor` (or record in `GlobalExceptionHandler`) that writes an audit row with `outcome=DENIED/FAILED` for `AccessDeniedException`/`AuthenticationException`/`ApiException` on mutating endpoints. Explicitly record `auth.login.failed` and `auth.login.locked` in `AuthController.login` on each failure branch (with email/actor and IP) via a direct `AuditWriter.record` call rather than relying on the after-returning interceptor. Add an explicit `outcome` column to `audit_log` so success and failure are first-class.

## [SEC-006] Append-only enforcement is bypassable: application connects as the audit table's owning DB role

- **Severity:** HIGH
- **Dimension:** audit-integrity
- **File(s)/location:** `control-panel-api/src/main/resources/db/changelog/changes/08-audit.sql` â€” `audit_log_block_modifications()` function and `audit_log_no_update`/`audit_log_no_delete` triggers (lines 20-34, function declared `CREATE OR REPLACE` at line 21). Datasource role: `application.yml` `DB_USER` default `cp` / `DB_PASS` default `cp` (lines 6-7), Liquibase runs as same role (lines 15-16); `docker-compose.yml` lines 5-7/39-40 (`POSTGRES_USER`/`POSTGRES_PASSWORD = cp/cp`). `AuditLog.java` line 20 (`@Immutable`). UI claim: `admin-ui/src/pages/AuditPage.tsx` line 53 ("Immutable, append-only.").

- **Description:** Tamper resistance relies entirely on two BEFORE UPDATE/DELETE triggers calling `audit_log_block_modifications()`. The Hibernate `@Immutable` annotation on `AuditLog` is only an ORM-level hint and provides no DB-level protection. Both the Liquibase migrations and the running application use the same DB role (`cp`/`cp`), which is the owner of `audit_log` and the trigger function. In PostgreSQL a table/trigger owner can `ALTER TABLE audit_log DISABLE TRIGGER ALL`, `DROP TRIGGER`, or `CREATE OR REPLACE FUNCTION audit_log_block_modifications()` to a no-op, then `UPDATE`/`DELETE` freely and re-enable. There is no separate least-privilege application role with INSERT/SELECT-only and no ownership. All 13 SQL migrations were checked: zero `CREATE ROLE`/`CREATE USER`/`GRANT`/`REVOKE`/`ALTER DEFAULT PRIVILEGES`/`ALTER TABLE ... OWNER`/`SET ROLE` statements exist. Therefore any path that yields the application's DB credentials or SQL execution as that role (SQL injection, compromised app host, leaked DB creds) defeats the immutability guarantee entirely, while the UI continues to advertise the log as "Immutable, append-only." The trigger approach does block accidental ORM writes and tampering by a hypothetical lower-privilege role, but since the runtime role IS the owner, the append-only guarantee is bypassable. High (not critical) because exploitation requires SQL execution as the app role (a real precondition, not trivial remote), but the impact is silent, complete destruction of the tamper-evident audit trail.

- **Attack scenario:** An attacker who obtains the app's DB credentials (default `cp`/`cp`, or via a SQLi sink) runs: `ALTER TABLE audit_log DISABLE TRIGGER audit_log_no_delete; DELETE FROM audit_log WHERE actor_user_id = '<attacker>'; ALTER TABLE audit_log ENABLE TRIGGER audit_log_no_delete;` â€” erasing all trace of their actions, while the UI continues to advertise the log as immutable.

- **Recommended fix:** Run the application under a dedicated, non-owning DB role granted only INSERT and SELECT on `audit_log` (no UPDATE/DELETE/TRUNCATE, no table ownership, no trigger privileges). Keep DDL/migrations under a separate privileged role used only by Liquibase. Consider `REVOKE TRUNCATE` and shipping audit rows to an append-only external sink (SIEM / WORM storage) for defense in depth. Do not rely on owner-modifiable triggers as the sole integrity control.

## [SEC-007] Session tokens cannot be revoked; logout is a no-op (no denylist / jti store), so suspended/disabled users keep full access for the 12h TTL

- **Severity:** HIGH
- **Dimension:** auth-session
- **File(s)/location:** `control-panel-api/src/main/java/com/example/cp/auth/AuthController.java` â€” `logout()` lines 118-126 (audit + 204 only; inline comment line 120: "Stateless JWT â€” client just drops the token"); `SessionTokenService.java` â€” `issue()` line 68 (`jwtID(UUID.randomUUID().toString())`, never persisted), `parse()` lines 81-110 (validates signature + expiry only), default TTL `PT12H` line 38; `JwtAuthFilter.java` â€” `doFilterInternal()` lines 41-87 (never reloads the User; never references `User.Status`). Config: `application.yml` line 37 `session-ttl: PT12H`; `SecurityConfig` line 86 (`/api/**` `.authenticated()`). `User.Status` `{ACTIVE, SUSPENDED, DELETED}` (`User.java` line 27), checked ONLY at login (`AuthController` line 94).

- **Description:** The admin session JWT is fully stateless. A unique `jwtID` (jti) is generated in `SessionTokenService.issue()` but never persisted, and `parse()`/`JwtAuthFilter` never consult any denylist, token-version, or session store. `/api/v1/auth/logout` only writes an audit entry and returns 204 â€” it does not invalidate anything. With `session-ttl` defaulting to `PT12H`, there is no way to invalidate a token before natural expiry. Critically, `JwtAuthFilter` never re-loads the User during request authentication, so it never observes that the account became SUSPENDED or DELETED â€” login is the only place status is checked. A full grep of control-panel-api for jti/denylist/blacklist/revoke/token-version/session-store/invalidate returns matches ONLY in the license-token (`LicenseToken`, `LicenseRevocationService`, `CrlController`) and api-key (`ApiKey`, `ApiKeyService`) subsystems â€” unrelated to admin sessions. No session/token denylist or token-version field exists.

- **Attack scenario:** An admin laptop is compromised and the bearer token is exfiltrated, or an employee is offboarded and their account set to SUSPENDED/DELETED. Security disables the account and the user clicks "logout", but the previously issued token remains fully valid for up to 12 hours and continues to authenticate every `/api/**` request (issuing licenses, revoking licenses, managing orgs, etc.). There is no server-side action that can stop it.

- **Recommended fix:** Introduce server-side revocation: persist a session record keyed by jti (or a per-user `token_version` / token-not-valid-before timestamp). On logout and on user suspend/delete/password-reset, add the jti to a denylist (e.g. Redis with TTL = remaining token lifetime) or bump `token_version`. In `JwtAuthFilter`/`parse()`, reject tokens whose jti is denylisted or whose `token_version`/`iat` predates the user's invalidation timestamp. Also reduce the 12h TTL and add refresh-token rotation if longer sessions are required.

## [SEC-008] super_admin and authorities are trusted from the token claims and never re-validated against current user state

- **Severity:** HIGH
- **Dimension:** auth-session
- **File(s)/location:** `control-panel-api/src/main/java/com/example/cp/auth/JwtAuthFilter.java` â€” `doFilterInternal()` lines 61-73 (line 61 `if (parsed.superAdmin())`; lines 64-65 comma-joined `authorities` claim; lines 66-68 fresh load only when baked claim empty; line 73 builds `AuthenticatedUser`). `SessionTokenService.issue()` lines 64-65 (bakes in super_admin + authorities), `parse()` lines 98-99 (reads `super_admin`). `AuthenticatedUser.hasAuthority()` lines 17-19 (short-circuits to true when superAdmin). `User.Status` `{ACTIVE, SUSPENDED, DELETED}`; `AuthoritiesLoader.authoritiesFor` (queries permissions only, lines 26-35). Chain STATELESS: `SecurityConfig` line 65. Default TTL `PT12H`.

- **Description:** `JwtAuthFilter` builds the request principal directly from the token: `parsed.superAdmin()` (line 61) and, for non-super-admins, the comma-joined `authorities` claim baked in at login (lines 64-65). For super admins it re-resolves authorities each request, but it still trusts the `super_admin` boolean from the token itself rather than reloading the User; the per-request DB read (line 63) is gated on the already-trusted `super_admin` boolean, so it does not mitigate demotion. `AuthenticatedUser.hasAuthority()` short-circuits to true whenever superAdmin is set (`return superAdmin || (authorities != null && authorities.contains(code))`), so the entire RBAC layer is bypassed based solely on a self-contained, 12h-valid token claim. The `User` entity exposes a mutable `superAdmin` field with a `@Setter`, but the filter never compares against `User.isSuperAdmin()` from the DB. For regular users, removing a permission grant (`user_roles`/`role_permissions`) is not re-checked, so the old permission set persists until expiry (a user who had real permissions at login keeps the stale set; only an empty baked claim triggers a fresh load). The filter never loads the User, so a suspended or deleted user with a valid unexpired token still authenticates with full prior privileges. The chain is STATELESS with no token blacklist or token/session version. Because there is no revocation (see SEC-007), demoting a super admin, suspending the account, or removing a permission grant has no effect until the token naturally expires.

- **Attack scenario:** A super admin is demoted (super_admin flag cleared) after a security incident, or a regular admin has the `license.revoke` permission removed. Both users continue to exercise the old privilege level for up to 12 hours using their existing token, since the filter reads `super_admin`/`authorities` from the token and never reloads the User or the user's current role/permission grants.

- **Recommended fix:** Do not trust the `super_admin` claim or frozen authorities for authorization decisions over the token lifetime. Reload the User (verify `status == ACTIVE` and the current `super_admin` flag) and/or re-resolve permissions on each request, or gate it behind a short-lived cache plus the revocation/token-version mechanism (SEC-007). At minimum, re-check `User.status == ACTIVE` in the filter so suspended/deleted accounts are rejected immediately.

## [SEC-009] In-organization privilege escalation: an ADMIN can add a member with the OWNER role

- **Severity:** HIGH
- **Dimension:** authz-rbac
- **File(s)/location:** `control-panel-api/src/main/java/com/example/cp/orgs/OrgController.java` â€” `addMember()`, `POST /api/v1/orgs/{orgId}/members`, lines 65-77 (`@PreAuthorize("@orgAccess.hasRole(#orgId, 'ADMIN')")`); `AddMemberRequest` lines 100-102 (only `@NotBlank` on role); role parse at line 71 (`OrgMember.Role.valueOf(body.role().toUpperCase())`). `OrgAccessChecker.hasRole` lines 29-41 (rank-based: ADMIN=3, OWNER=4). `OrgMember.java:29` (OWNER enum constant). `OrgService.addMember` lines 95-117 (persists with no rank guard; `userRepository.findByEmail` at line 101). Transfer-owner gate: `OrgController` lines 86-87.

- **Description:** `addMember` only requires ADMIN rank on the org (`@orgAccess.hasRole(#orgId,'ADMIN')`, satisfied by ADMIN=3 or OWNER=4 per `OrgAccessChecker.rank()`), but the request body lets the caller specify ANY role string, parsed into `OrgMember.Role.valueOf(...)` which accepts OWNER. There is no check that the actor's rank is greater than or equal to the rank being granted. `hasRole` only enforces a minimum actor rank, never compares it to the assigned role; the body validates only `@NotBlank` on the role string with no enum/value restriction; and `OrgService.addMember` persists whatever role was passed with NO actor-rank-vs-granted-role comparison. - **Attack scenario:** A customer signs up, creates an org (gets OWNER), then POSTs an OIDC SSO provider with config `{"issuer":"http://169.254.169.254/latest/meta-data/iam/security-credentials"}` (or a SAML provider with `metadataUrl` pointing at `http://localhost:8080/actuator`, an internal Redis/Postgres admin, or a Kubernetes API at `https://10.0.0.1`). - **Description:** The ingest endpoint authorizes with only `SecurityUtils.requireUser()`, which returns the principal for ANY authenticated session â€” any console user (regardless of org) or any API-key holder â€” and does not check authorities or org membership.
